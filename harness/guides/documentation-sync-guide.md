## 1. 概述

> **重要**: 随着 Harness Engineering 体系的完善，CLAUDE.md 已废弃，所有上下文信息迁移至 harness/ 目录管理。

---

## 2. 重大更新定义

以下情况属于"重大更新"，必须在完成后24小时内同步更新相关文档：

| 更新类型 | 具体范围 | 判定依据 |
|----------|----------|----------|
| **功能模块实现** | 新增/修改/删除功能模块 | 代码提交涉及新Controller/Service/组件 |
| **系统架构调整** | 技术栈变更、架构模式调整 | pom.xml、配置文件变更 |
| **API接口变更** | 新增/修改/删除API接口 | Controller层代码变更 |
| **UI/UX设计优化** | 页面布局、交互流程变更 | 前端组件/页面变更 |
| **技术栈升级** | 依赖版本升级、新技术引入 | package.json/pom.xml版本变更 |
| **性能改进** | 关键性能优化措施 | 性能测试报告或优化代码提交 |
| **数据库变更** | 表结构、字段变更 | schema.sql或实体类变更 |


## 5. 标准化更新操作步骤

### 步骤1: 信息收集
- [ ] 收集本次更新的代码变更清单
- [ ] 整理变更影响范围
- [ ] 记录关键实现细节

### 步骤2: 内容编写
- [ ] 确定需要更新的文档列表
- [ ] 按照规范格式编写更新内容
- [ ] 更新版本号和最后更新时间

### 步骤3: 审核验证
- [ ] 自查更新内容完整性
- [ ] 确认与其他文档无冲突
- [ ] 验证技术细节准确性

### 步骤4: 版本控制
- [ ] 通过Git提交文档更新
- [ ] 编写规范的提交信息
- [ ] 推送到远程仓库

## 6. 同步内容规范格式

### 6.1 变更记录格式

每次文档更新必须在文档末尾的"变更记录"部分添加记录：

```markdown
| 日期 | 版本 | 变更内容 | 责任人 | 关联任务ID |
|------|------|----------|--------|------------|
| YYYY-MM-DD | Vx.x | [变更类型] 变更摘要 | 姓名 | TASK-xxx |
```

### 6.2 变更类型标识

| 标识 | 说明 |
|------|------|
| [新增] | 新增功能或文档章节 |
| [修改] | 修改现有内容 |
| [删除] | 删除功能或内容 |
| [优化] | 性能或体验优化 |
| [修复] | 问题修复 |
| [文档] | 纯文档更新 |

### 6.3 版本号规则

- **主版本号 (Major)**: 重大架构变更、不兼容更新
- **次版本号 (Minor)**: 功能新增、兼容性更新
- **修订号 (Patch)**: 问题修复、小优化

示例: V1.0 → V1.1 (功能新增) → V1.1.1 (问题修复)

## 8. 异常情况处理

### 8.1 文档冲突解决

```
发现冲突 → 分析冲突原因 → 协商解决方案 → 合并更新 → 重新验证
```

**处理原则**:
1. 以最新代码实现为准
2. 保留双方有效信息
3. 标注争议点待确认

### 8.2 信息丢失恢复

| 场景 | 恢复方案 |
|------|----------|
| 误删文档内容 | Git回退到上一版本 |
| 提交信息丢失 | 查看Git历史记录恢复 |
| 版本混乱 | 重新梳理版本历史，标注正确版本 |

### 8.3 应急预案

1. **文档损坏**: 从Git历史恢复最近正确版本
2. **同步遗漏**: 补充变更记录，标注补录时间
3. **版本冲突**: 召开同步会议，统一版本状态
