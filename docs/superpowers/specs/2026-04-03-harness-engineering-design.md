## 1. 项目概述

### 1.1 改造背景

本项目旨在将现有的报表数据处理平台改造为符合 Harness Engineering（驾驭工程）模式的项目，使 AI Agent 能够在约束环境中稳定、可靠地参与开发工作。

### 1.2 改造目标

| 目标 | 描述 |
|------|------|
| **知识体系化** | 建立 AI Agent 可理解的知识地图和文档体系 |
| **状态结构化** | 使用结构化文件管理任务和进度，抗模型腐化 |
| **验证自动化** | 完善测试和代码检查，建立快速反馈循环 |
| **渐进式改造** | 保持现有功能不变，逐步引入 Harness 元素 |

### 1.3 改造范围

| 范围 | 说明 |
|------|------|
| **核心改造** | AGENTS.md、任务管理、文档体系 |
| **保留不变** | 所有现有业务代码和功能 |
| **文档整合** | 统一 superpowers 和 harness 文档体系 |

## 2. 架构设计

### 2.1 整体架构

```
report-front/
├── AGENTS.md                    # AI Agent 知识地图（核心入口）
├── harness/                     # Harness 状态管理
│   ├── tasks.json              # 结构化任务清单（单一事实来源）
│   ├── progress-notes.md       # 进度记录
│   ├── session-history/        # 会话历史归档
│   └── templates/              # 文档模板
├── docs/
│   ├── superpowers/            # 主要文档区（保留现有结构）
│   │   ├── plans/              # 实施计划
│   │   └── specs/              # 技术规范
│   ├── harness/                # Harness 补充文档
│   │   ├── conventions/        # 编码规范
│   │   ├── guides/             # Agent 操作指南
│   │   └── reference/          # 参考资料
│   └── changes/                # 设计变更记录
├── report-backend/             # 后端代码（不变）
├── src/                        # 前端代码（不变）
└── ...                         # 其他现有文件（不变）
```

### 2.2 目录说明

#### AGENTS.md
- **用途**：AI Agent 的单一事实来源
- **原则**：渐进式披露，简洁导航，指向详细文档
- **规模**：100-200 行

#### harness/ 目录
- **tasks.json**：结构化任务清单，JSON 格式，抗模型腐化
- **progress-notes.md**：每次会话结束时更新的进度记录
- **session-history/**：归档历史会话，便于回溯
- **templates/**：文档模板，保持一致性

#### docs/ 目录整合策略
- **docs/superpowers/**：继续作为主要文档区，plans/ 和 specs/ 符合 Harness 标准
- **docs/harness/**：存放 Harness 专用补充文档
- **docs/changes/**：设计变更记录

### 3.2 tasks.json 任务清单

#### JSON Schema

```json
{
  "version": "1.0",
  "lastUpdated": "2026-04-03",
  "tasks": [
    {
      "id": "T-001",
      "title": "创建 AGENTS.md 知识地图",
      "description": "创建 AI Agent 的单一事实来源文档，包含项目概述、目录导航、架构约束等",
      "status": "pending",
      "priority": "high",
      "dependsOn": [],
      "createdAt": "2026-04-03",
      "completedAt": null,
      "assignee": "AI Assistant"
    }
  ]
}
```

#### 状态枚举

| 状态 | 说明 |
|------|------|
| `pending` | 待开始 |
| `in_progress` | 进行中 |
| `completed` | 已完成 |
| `blocked` | 被阻塞 |

#### 优先级枚举

| 优先级 | 说明 |
|--------|------|
| `high` | 高 |
| `medium` | 中 |
| `low` | 低 |

#### 操作规则

- **不可删除**：任务永远保留，只改变状态
- **不可重排序**：保持任务创建顺序
- **只追加**：新任务添加到数组末尾
- **状态转换**：`pending` → `in_progress` → `completed`

### 3.3 会话协议

#### 标准会话流程

1. **Orient（定位）**
   - 读取 progress-notes.md
   - 读取 tasks.json
   - 查看最近 git history

2. **Setup（准备）**
   - 确保开发环境正常
   - 验证现有功能

3. **Select Task（选择任务）**
   - 选择最高优先级的 pending 任务
   - 更新任务状态为 in_progress

4. **Implement（实施）**
   - 专注完成单个任务
   - 遵循架构约束

5. **Test（测试）**
   - 运行相关测试
   - 验证功能正常

6. **Record（记录）**
   - 更新 progress-notes.md
   - 更新任务状态为 completed
   - Git 提交

## 4. 验证机制

### 4.1 测试策略

#### 现有测试增强

| 层级 | 说明 |
|------|------|
| **单元测试** | 后端核心服务补充单元测试 |
| **组件测试** | 前端组件添加测试 |
| **集成测试** | API 接口测试完善 |
| **E2E 测试** | 端到端测试基础框架 |

### 4.2 代码规范检查

#### 前端（ESLint）
- 已存在配置，优化规则
- 配置 husky + lint-staged 提交前检查

#### 后端（Checkstyle/SpotBugs）
- 添加代码风格检查
- 潜在 Bug 检测

### 4.3 Git 工作流

#### 分支策略

```
main/master (保护分支)
  └── develop (集成分支)
        ├── feature/T-001-agents-md
        ├── feature/T-002-tasks-json
        └── ...
```

#### 提交信息规范（Conventional Commits）

```
<type>(<scope>): <description>

[可选的详细说明]

关联任务: T-001
```

**Type 类型**：
- `feat`: 新功能
- `fix`: 修复
- `docs`: 文档
- `refactor`: 重构
- `test`: 测试
- `chore`: 构建/工具
