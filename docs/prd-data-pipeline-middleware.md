## 1. Problem Statement（问题陈述）

### 背景
公司内部存在两条 RPA 数据链路：上游 RPA 从各个业务系统爬取报表数据并上传至 FTP，下游 RPA 定期从 FTP 获取压缩包并通过 WebUI 逐条上传入库。两段 RPA 之间缺乏一个自动化的数据转换与打包中间层。

### 核心痛点

1. **非标表格解析困难**：上游报表多为"中国风"表格——合并单元格、多行表头、中文列名、非 A1 起始，结构差异极大（N 张表 = N 种格式），下游无法直接消费
2. **缺失自动化转换**：无标准化的字段映射、数据清洗和打包能力，开发人员需频繁手工介入
3. **数据传输低效**：下游 RPA 通过 WebUI 逐文件上传，散乱文件浪费大量等待时间；缺少批量打包能力
4. **无监控与追溯**：字段变更、解析失败难以感知，易造成数据漏传

### 影响现状
- 日均几十张报表流转，每张格式不同，靠临时脚本处理
- FTP 目录无归档管理，文件堆积混乱
- 下游上传效率低，端到端链路不可靠

## 3. Non-Goals（V1 不做的事）

- ❌ 不提供拖拽式可视化编辑器（使用表单配置 JSON）
- ❌ 不直接连接数据库抽取数据（仅 FTP → FTP 的 Excel 文件流转）
- ❌ 不替代上游或下游 RPA（不做数据爬取，不做数据入库）
- ❌ 不支持非 Excel 格式（PDF、Word 等不在范围内）
- ❌ 不在此阶段做深度业务逻辑校验，仅做字段级清洗转换

## 4. Core User Stories（核心用户故事）

### Primary Persona：数据开发工程师
负责维护报表数据链路，熟悉 FTP 操作、了解源表结构，能阅读并配置字段映射规则。日常检查处理状态和处理告警。

---

### Story 1：FTP 监控与文件匹配
**作为** 数据开发工程师  
**我想要** 在管理界面上配置 FTP 监控目录和文件名匹配规则  
**以便** 系统能自动识别新增的报表文件并匹配对应的解析模板

**验收标准**：
- [ ] Given 已配置 FTP 连接信息（地址、端口、账号、密码）和监控目录，When 系统执行定时扫描，Then 列出目录下所有文件
- [ ] Given 已配置文件名匹配模式（如 `report_*.xlsx`），When 目录下存在符合规则但未曾处理的文件，Then 系统识别该文件并触发处理
- [ ] Given 文件名中含日期（如 `sales_report_20260426.xlsx`），When 系统进行文件名匹配，Then 自动提取日期作为 pt_dt 分区字段值
- [ ] Given 文件名不含可识别日期，When 系统尝试解析，Then 按配置的默认策略处理（如取文件修改时间或报错告警）

---

### Story 2：中国风表格解析与字段映射
**作为** 数据开发工程师  
**我想要** 配置报表的起始行列偏移和字段映射规则（按列名 / 按列序号）  
**以便** 解析各种非标准布局的中国风报表并输出为标准字段名 Excel

**验收标准**：
- [ ] Given 报表有效数据从非 A1 开始（如第 4 行第 2 列），When 配置起始行号和起始列号，Then 系统从指定偏移位置开始解析
- [ ] Given 配置了按列名映射（源列名 → 目标列名），When 列顺序发生变化但列名未变，Then 系统仍能正确映射（位置无关）
- [ ] Given 配置了按列序号映射（源第 N 列 → 目标列名），When 列名变更但位置未变，Then 按序号提取数据
- [ ] Given 源表存在重复列名，When 解析，Then 按配置策略区分（自动加后缀编号 _1, _2 或按序号标识）
- [ ] Given 源表的被映射字段缺失（列名或序号均不匹配），When 系统解析，Then 触发告警并跳过该文件

---

### Story 3：列级异常值清洗
**作为** 数据开发工程师  
**我想要** 按列配置异常值替换规则  
**以便** "-"、"N/A"、"null" 等源系统占位符能正确转换，避免下游入库类型报错

**验收标准**：
- [ ] Given 配置某列的 "-" → "0"，When 系统处理到该列 "-" 值，Then 自动替换为 0
- [ ] Given 配置某列的自定义转换规则（如 "空" → ""），When 遇到匹配的异常值，Then 按规则转换
- [ ] Given 某列未配置清洗规则且遇到不可自动转换的值，When 处理，Then 保留原值并在处理记录中标记警告

---

### Story 4：批量打包与配置表生成
**作为** 数据开发工程师  
**我想要** 同一批次处理完成的文件自动加入压缩包，并生成一张配置 Excel 表记录每文件的目标信息  
**以便** 下游 RPA 能一次获取多个文件，提升 WebUI 上传效率

**验收标准**：
- [ ] Given 一个扫描周期内 N 个文件处理完成，When 系统生成压缩包，Then N 个标准化 Excel 全部加入同一个压缩包
- [ ] Given 压缩包内有若干标准 Excel，When 系统生成配置 Excel 表，Then 配置表中每文件一条记录，至少包含：源文件名、库名、表名、字段映射（JSON）、是否分区、分区字段、分区字段值
- [ ] Given 压缩包生成完毕，When 系统操作 FTP，Then 压缩包上传至指定目录供下游 RPA 消费
- [ ] Given 压缩包已上传成功，When 系统检查，Then 进入源文件归档流程

---

### Story 5：源文件归档
**作为** 数据开发工程师  
**我想要** 处理成功后的源文件自动移动到归档目录  
**以便** FTP 监控目录保持整洁，避免文件被重复扫描打包

**验收标准**：
- [ ] Given 文件处理成功且已打包上传，When 系统完成流程，Then 源文件自动移动到预设归档目录
- [ ] Given 归档目录已有同名文件，When 系统归档，Then 自动追加时间戳后缀避免覆盖
- [ ] Given 文件处理失败（被跳过），When 系统判断，Then 源文件保留原地或移动到失败目录供人工排查

---

### Story 6：告警与可视化
**作为** 数据开发工程师  
**我想要** 在管理界面查看解析异常告警和处理记录  
**以便** 快速感知上游字段变更并及时修复配置

**验收标准**：
- [ ] Given 某文件字段映射失败，When 系统触发告警，Then 管理界面告警列表中显示该条告警（含文件名、时间、失败原因）
- [ ] Given 一条文件处理失败，When 系统跳过该文件，Then 继续处理下一个文件（不中断批量流程）
- [ ] Given 我打开管理界面，When 查看处理记录列表，Then 能看到每次处理的文件名、状态（成功/失败/跳过）、处理时间、所属批次

## 5. Solution Overview（方案概述）

### 系统定位
**FTP 报表数据转换中间件**——位于上游 RPA（数据采集）和下游 RPA（数据入库）之间的自动化管道。核心职责：FTP 新文件检测 → 中国风表格解析 → 字段映射+数据清洗 → 标准化 Excel 生成 → 批量打包含配置表压缩包 → FTP 回传 → 源文件归档。

### 核心流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ FTP 扫描 │ → │ 文件匹配 │ → │ 表格解析 │ → │ 字段映射 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                        ↓
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 源文件归档│ ← │ 压缩包上传│ ← │ 批量打包 │ ← │ 数据清洗 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

### 系统架构（两大子系统）

**子系统 A：数据处理引擎（后端）**
- 定时扫描 FTP 监控目录，按配置的目录+文件名模式识别新文件
- 按报表规则从指定行/列偏移开始解析 Excel
- 执行字段映射（按列名优先，按列序号兜底）
- 处理重复列名（自动编号去重）
- 执行列级异常值清洗规则
- 生成标准化 Excel 文件
- 将同一批次处理结果批量加入压缩包
- 自动生成包内配置 Excel 表
- 压缩包上传至 FTP 指定目录
- 源文件归档

**子系统 B：管理端 Web 应用（前端 + 后端 API）**
- FTP 监控任务 CRUD
- 报表解析规则 CRUD（包含起始行列、字段映射、清洗规则、目标库表信息）
- 处理记录查询（成功 / 失败 / 跳过）
- 告警列表查询

### Key Design Decisions

| 决策 | 方案 | 理由 | 取舍 |
|------|------|------|------|
| 映射模式 | 列名优先 + 列序号兜底 | 列名映射容忍列顺序变化；序号映射在列名变更时仍可用 | 两种模式需要额外配置字段 |
| 打包策略 | 单批次多文件合并入包 | 下游 WebUI 上传慢，批量打包提升数十倍效率 | 合并后若单个文件出问题，排查需从包内定位 |
| 归档策略 | 成功后立刻移动归档 | 保证每文件仅处理一次；FTP 目录保持整洁 | 如果归档后下游消费异常需重新生成，需人工介入 |
| 配置方式 | 表单项 + JSON | 兼顾配置效率和灵活性；避免 V1 花费过多时间做可视化编辑器 | 开发人员需了解 JSON 基本语法 |
| 告警方式 | 网页端面板 | V1 轻量实现，不做额外通道对接 | 非实时推送，依赖主动查看 |

## 6. Technical Considerations（技术考量）

### 技术栈选型

| 层级 | 选型 | 备注 |
|------|------|------|
| 前端 | Vue 3 | 内部 Web 应用 |
| 后端 | SpringBoot | RESTful API |
| Excel 解析 | Apache POI / EasyExcel | 流式读取防止 50MB 文件 OOM |
| 压缩打包 | Java ZipOutputStream | 标准库，零额外依赖 |
| FTP 操作 | Apache Commons Net | 成熟的 FTP 客户端库 |
| 任务调度 | Spring @Scheduled / Quartz | 按配置间隔（如 5 分钟）触发扫描 |
| 数据存储 | MySQL / H2 | 配置持久化 + 处理记录存储 |

### 关键外部依赖

| 依赖项 | 说明 | 对接方 | 风险 |
|--------|------|--------|------|
| FTP Server 稳定性 | 文件读写全链路依赖 FTP | 运维 | Medium |
| 上游 RPA 不改 FTP 目录结构 | 目录+文件名匹配依赖固定结构 | RPA 团队 | Low |
| 下游 RPA 压缩包消费 | 压缩包格式和内容需对齐 | RPA 团队 | Medium |

### Known Risks & Mitigations

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 复杂合并单元格 POI 解析失败 | Medium | High | Sprint 0 做解析 Spike；若不可行改用 Python 脚本辅助 |
| 50MB 大文件 OOM | Low | Medium | EasyExcel 流式读取 + JVM 内存配置 |
| FTP 连接中断致数据丢失 | Low | High | 异常时源文件不做归档，保留原位待重试；增加重试机制 |
| 列名重复导致映射冲突 | Medium | Low | 配置阶段校验重复列名并强制配置区分策略 |

### Open Questions（开发前必须解决）

- [ ] POI/EasyExcel 对多行合并单元格表头的解析能力是否满足当前中国风报表场景？—— Owner: Tech Lead —— Deadline: Sprint 0 Spike
- [ ] 压缩包内配置 Excel 的字段规格与下游 RPA 的解析逻辑是否完全对齐？需安排一次对齐会议。—— Owner: Adam —— Deadline: Sprint 1 启动前
- [ ] 是否需要历史处理记录定期清理策略？存储空间规划。—— Owner: Tech Lead —— Deadline: 技术设计 Review
